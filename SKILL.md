# 中文转文字稿整理

## 核心目标

将原始转写整理为“扩展完整通顺版”：

- 保留原稿主体信息量，不改写成课程提要或会议摘要。
- 修复语序、断句、重复、指代和明显语音识别错误。
- 保留说话人的推导过程、连续追问、观点分歧、反例和案例细节。
- 只重构能够由上下文可靠恢复的内容；不猜造人名、地名、数字、案名或结论。
- 保持原有观点，不替发言人补充未经原稿支持的新学说或事实。

详细尺度与例子见 [references/quality-standard.md](references/quality-standard.md)。开始改稿前必须阅读该文件。

## 工作流

### 1. 找到并检查输入

1. 查找用户指定的原稿；名称不完全一致时按日期、主题和文件格式匹配。
2. 保留原文件，不直接覆盖。
3. 提取全文并统计：
   - 段落数和中文字符数；
   - 标题层级；
   - `【老师】`、`【同学】`、`【主持人】` 等角色标记；
   - 明显乱码、重复词、断裂句和疑似错识专名。
4. 如果存在多个版本，以最原始、信息量最大的版本为事实底稿；已整理版本只用于参考结构和措辞。

### 2. 先建立内容地图

按原稿顺序列出每个主题的：

- 核心问题；
- 案例事实或材料；
- 发言人的主要结论；
- 支撑理由；
- 追问、反例和不同意见；
- 暂未解决的问题。

不要在此阶段压缩内容。内容地图用于防止遗漏，不是最终摘要。

### 3. 按段落重构

对每一段执行以下顺序：

1. 确定说话人和本段中心。
2. 删除无意义口头填充，如连续的“然后”“就是说”“对对对”，但保留具有互动意义的简短回应。
3. 合并同一说话人紧邻的机械重复；保留重复中新增的理由或例子。
4. 调整主谓宾和时间顺序，补足可从上下文确定的省略成分。
5. 把混在一段中的多个说话人拆开。
6. 把连续追问保留为问答链，不要直接替换成一个结论。
7. 对无法可靠恢复的局部采用保守表达；若关键事实无法判断，标记为`【转写存疑】`，不要猜测。

### 4. 控制详尽程度

默认采用“扩展完整通顺版”，不是摘要版：

- 一般以原稿中文字符量的约 65%–85% 为参考区间。
- 原稿乱码和纯重复很多时可低于 65%，但必须保留所有可理解的实质内容。
- 原稿本身较整洁时应接近 80%–95%。
- 不通过空泛解释、常识扩写或自行补充知识来凑字数。
- 每个一级主题通常保留事实、观点、理由、反驳和小结，而不只是结论。

若用户说“再详细一点”，必须回到原始底稿寻找被遗漏的可辨认内容，不要仅扩写上一版的概括句。

### 5. 法律、学术和专业稿件

- 保留法条号、案名、金额、时间、裁判层级和构成要件。
- 发现明显错识时，优先依据同一稿件上下文修正。
- 对可能随时间变化或高风险的法律信息，使用权威来源核对；将核对用于纠错，不擅自增加原课堂未讨论的新内容。
- 区分老师明确结论、同学观点和课堂假设，不将讨论性说法改成确定事实。
- 对专名无法确认时使用概括表达，不制造看似精确的错误名称。

### 6. 输出 DOCX

处理 DOCX 时同时使用文档处理 Skill 的要求：

1. 延续原文标题和基础样式。
2. 保留真实的标题层级和发言标记。
3. 另存为：
   - `<原名>-扩展完整通顺版.docx`，或
   - 用户指定名称。
4. 不在最终文件中保留内部注释、脚本标记或占位符。
5. 使用文档渲染工具生成逐页 PNG，检查所有页面是否截字、重叠、缺字、断页异常。

### 7. 审计

使用：

```bash
python scripts/audit_transcript.py ORIGINAL.docx REVISED.docx
```

检查并人工复核：

- 信息量保留率是否合理；
- 一级标题是否丢失；
- 角色标记是否仍然存在；
- 是否残留重复词、连续口头词或`【转写存疑】`；
- 是否把讨论误写成摘要；
- 是否有新增但无原稿依据的事实。

审计脚本只提供信号，不能代替逐段语义检查。

## 禁止事项

- 不把两万字逐字稿压缩成五六千字提要，除非用户明确要求摘要。
- 不为了通顺删除同学的反对意见、老师的追问或案例计算过程。
- 不把多个立场融合成一个“标准答案”。
- 不根据常识臆造录音中听不清的人名、案例名称或数字。
- 不覆盖原文件。
- 不只检查文本内容而跳过 DOCX 页面渲染。

## 交付

最终只链接交付文件，并简要说明：

- 输出采用“扩展完整通顺版”；
- 页数或字数；
- 已保留原文件；
- 已完成页面检查。
